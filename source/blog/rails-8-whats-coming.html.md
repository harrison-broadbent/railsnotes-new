> Work on Rails 8 is starting, so I dug through the [official GitHub milestone](https://github.com/rails/rails/milestone/87) to break down all the upcoming goodies for you, right here in this article. There's plenty coming, including Solid Queue, Solid Cache and Kamal becoming defaults, better support for PWAs, an official LSP, an authentication generator and (much) more!

It's a new year, and the official [Rails 8 GitHub milestone](https://github.com/rails/rails/milestone/87) is live! This milestone gives us a great idea of the new stuff coming in Rails 8; I've done some digging and summarized all the exciting new goodies in this article.

With [Rails 8 slated for release sometime in 2024](https://twitter.com/dhh/status/1739776383637959000)\*, I think this article is a good way to get ahead (and excited!) about all the new stuff coming our way.

**The broad brushstrokes of Rails 8 center around simplifying the stack, leaning into PWA's, adding simple first-party solutions to things like authentication, and improving developer experience** — a lot of these additions seem driven by the [ONCE](https://once.com/) products, so the focus on simplification makes sense.

Let's dive in!

> \*Update: In a [recent blog post](https://world.hey.com/dhh/rails-world-and-rails-8-in-2024-c7b090ba), DHH confirmed that Rails 8 will be released **"...by the time Rails World 2024 rolls around"**. [Rails World 2024](https://rubyonrails.org/world/) is scheduled for September 26 & 27, 2024.
>
> So, if you're wondering about the Rails 8 release date, I'm thinking Rails 8 will be released on (or before) September 26, 2024.

## Solid Queue and Solid Cache as defaults (plus a DB-backed ActionCable adapter)

[Solid Queue](https://dev.37signals.com/introducing-solid-queue/) and [Solid Cache](https://dev.37signals.com/solid-cache/) were released with great fanfare towards the end of 2023, and they're slated to become the default queue and cache implementations for Rails 8.

This is a pretty big shift! Until now, job queues and caching in Rails both relied on in-memory stores (think Redis or Memcached). **These two new adapters** (alongside the new and currently unreleased [DB-backed ActionCable adapter](https://github.com/rails/rails/issues/50480) — _Solid Cable_, anyone?) **will let a Rails 8 app completely drop Redis/Memcached!**

Of course, you'll still be able to use Redis if you wish, and in a lot of situations you'll probably want to. But, **these DB-based adapters will let Rails 8 apps run without an in-memory store**, simplifying the stack, cutting down the number of services a production Rails 8 app needs, and saving some $$$ on cloud hosting costs.

> Related GitHub issues:
>
> - [Solid Queue should be the default Active Job backend for Rails 8](https://github.com/rails/rails/issues/50442)
> - [Solid Cache should be the default caching backend for Rails 8](https://github.com/rails/rails/issues/50443)
> - [Add DB-backed Action Cable adapter as new default in Rails 8](https://github.com/rails/rails/issues/50480)
>
> Note: There's a chance that [Mission Control (video)](https://www.youtube.com/watch?v=iqXjGiQ_D-A&t=3322s), the teased UI management panel for Solid Queue, could be included as part of Rails 8, but it's not guaranteed ([GitHub discussion](https://github.com/rails/rails/issues/50442#issuecomment-1871291464)).

## Progressive Web Apps (PWAs) are getting first-class support

> ... I have PWA tech as my #1 objective for Rails 8.
>
> — [DHH on Twitter](https://twitter.com/dhh/status/1742643733638648213)

**Progressive Web Apps (PWAs) are getting much better support in Rails 8, with the inclusion of [Action Notifier, a new push notification framework](https://github.com/rails/rails/issues/50454), plus required PWA files being [generated by default](https://github.com/rails/rails/issues/50455).**

Action Notifier is the more exciting of the two, serving as an entirely new framework within Rails for managing push notifications (including managing [badge counters](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps/How_to/Display_badge_on_app_icon), those little icons indicating your hundreds of unread emails). Action Notifier will be analogous to Action Mailer, but strictly for push notifications.

This push (hah!) for better PWA support in Rails 8 seems strongly in line with DHH's [pro-PWA stance](https://world.hey.com/dhh/native-mobile-apps-are-optional-for-b2b-startups-in-2024-4c870d3e), driven in no small part, I imagine, by the [app store shennanigans](https://twitter.com/dhh/status/1742643733638648213) he's faced building [HEY](https://www.hey.com/).

> Related GitHub issues:
>
> - [Extract Action Notifier framework for push notifications](https://github.com/rails/rails/issues/50454)
> - [Default PWA files](https://github.com/rails/rails/issues/50455)

## Basic Authentication Generator (finally!)

It's been a long time coming, but Rails 8 looks set to finally include an authentication generator!

From the GitHub issue —

> We can teach Rails developers how to use the basic blocks [of authentication] by adding a basic authentication generator that essentially works as a scaffold, but for authentication.
>
> — DHH

With Rails 7.1 including some [big improvements to home-rolled authentication](https://rubyonrails.org/2023/10/5/Rails-7-1-0-has-been-released), and rate-limiting [recently getting merged](https://github.com/rails/rails/pull/50490) into Action Controller, **Rails now contains all the pieces needed for a simple authentication solution.**

The authentication coming in Rails 8 sounds like it will be relatively simple, far simpler than a heavyweight solution like [Devise](https://github.com/heartcombo/devise). It sounds like [passkeys won't be initially supported](https://github.com/rails/rails/issues/50446#issuecomment-1870702235) (but that might come later), and will [generate code directly](https://github.com/rails/rails/issues/50446#issuecomment-1869816754) rather than acting as a mountable engine.

In many cases, a solution like Devise may still be preferable, and we're still waiting for a pull request to materialize, but I'm excited! It's about time, and in the spirit of Rails 8, a simple native solution will simplify a large chunk of Ruby on Rails apps (particularly those running at smaller scales).

> Related GitHub issues:
>
> - [Add basic authentication generator](https://github.com/rails/rails/issues/50446)
> - [Add rate limiting to Action Controller via the Kredis limiter type](https://github.com/rails/rails/pull/50490)

## Improved linting and editor experience (including a default Rubocop!)

We've already covered a ton of great stuff coming in Rails 8, but the slated improvements to developer experience are some of my personal favorites.

**Rails 8 has two major upcoming features here — a [default Rubocop setup](https://github.com/rails/rails/issues/50456), and a [default LSP](https://github.com/rails/rails/issues/50453).**

As you'd expect, the [Rubocop discussion thread](https://github.com/rails/rails/issues/50456) was a bit contentious! Despite heavy support for [Standard](https://github.com/standardrb/standard), Rails 8 looks set to ship with an [Omakase Rubocop configuration](https://github.com/rails/rubocop-rails-omakase) instead. The intention here seems twofold — It's to serve as a basic, pared-back linter for Rails 8+ apps, and also a starting point for an in-house style guide (for individual projects or companies).

The default LSP excites me a lot — I hope it leads to a vastly improved editor/IDE experience for Ruby on Rails, ideally including things like autocompletion, hinting, jump-to-definitions and more. As DHH mentioned in the issue comment, Shopify is slowly building out the [ruby-lsp-rails](https://github.com/Shopify/ruby-lsp-rails) gem, alongside [ruby-lsp](https://github.com/Shopify/ruby-lsp) — it's unclear whether `ruby-lsp-rails` will become the Rails default, or whether the Rails default will be derived from it, but I'm excited!

Between both of these improvements, I'm hoping Rails 8 will deliver a big bump to developer experience & happiness, à la [the Rails doctrine](https://rubyonrails.org/doctrine#optimize-for-programmer-happiness).

> Related GitHub issues:
>
> - [Add (a very basic!!) Rubocop by default](https://github.com/rails/rails/issues/50456)
> - [Add a default LSP for better editor/IDE support in Rails 8](https://github.com/rails/rails/issues/50453)

## Streamlined deployment with Kamal and GitHub CI Actions

With Rails 8 [including Brakeman by default](https://github.com/rails/rails/issues/50501) alongside Rubocop lint rules, a [default GitHub CI action](https://github.com/rails/rails/issues/50502) was the next logical step, and it's coming in Rails 8.

**This new GitHub CI action will cover linting, running tests, scanning for vulnerabilities, and a Dependabot config** for keeping dependencies up to date. Here's the [merged PR](https://github.com/rails/rails/pull/50508/files) if you're interested, with the intention being to [double down on setting good habits for new apps](https://twitter.com/dhh/status/1741838676487737763).

**[Kamal](https://kamal-deploy.org/) is also going mainstream and will be included in all new Rails 8 apps as the [default deployment solution](https://github.com/rails/rails/issues/50441)**, in line with DHH's trend towards platform agnosticism, and away from large cloud providers and vendor lock-in. Of course, you'll be able to use other deployment methods (like [Dokku](https://railsnotes.xyz/blog/deploying-ruby-on-rails-with-dokku-redis-sidekiq-arm-docker-hetzner)), and the Kamal installation is itself very lightweight (a single `deploy.yml` file, and a line in your `Gemfile`).

I'm hoping that Kamal's inclusion in Rails 8 will force it to mature a bit (the [official docs](https://kamal-deploy.org/docs/installation) seem a bit sparse), but I'm excited to watch the tool develop. The inclusion of a default GitHub CI action is also handy and will dovetail nicely into most people's workflows.

> Related GitHub issues:
>
> - [Add GitHub CI workflow by default to new applications](https://github.com/rails/rails/issues/50502)
> - [Add Brakeman by default to new apps](https://github.com/rails/rails/issues/50501)
> - [Setup Kamal by default for new applications](https://github.com/rails/rails/issues/50441)

## Other goodies (Propshaft, structured logging, benchmarking, HTTP/2 and more!)

Finally, here's a dump of the leftover goodies I couldn't fit above —

- [Propshaft](https://github.com/rails/propshaft) is becoming the [default asset pipeline](https://github.com/rails/rails/issues/50444) for Rails 8+ apps, taking over from [Sprockets](https://github.com/rails/sprockets-rails), and leaning into the [no build future](https://world.hey.com/dhh/you-can-t-get-faster-than-no-build-7a44131c) envisioned by DHH.

- Rails 8 will include [structured JSON logging](https://github.com/rails/rails/issues/50452), to complement the existing developer logging, with the implementation extracted from 37Signals' and Shopify's in-house solutions.

- There are also plans to add an [official benchmarking tool](https://github.com/rails/rails/issues/50451), and a [lightweight Go proxy](https://github.com/rails/rails/issues/50479) which will sit in front of Puma and provide HTTP/2, public caching, `gzip` compression and more.

> Related GitHub issues:
>
> - [Make Propshaft the default asset pipeline in Rails 8](https://github.com/rails/rails/issues/50444)
> - [Add structured logging next to developer logging](https://github.com/rails/rails/issues/50452)
> - [Add a built-in benchmark tool](https://github.com/rails/rails/issues/50451)
> - [Add Airlock\* to Docker setup to get HTTP/2, X-Sendfile, Caching by default in Rails 8](https://github.com/rails/rails/issues/50479)
